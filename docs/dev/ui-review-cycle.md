# UI Review Cycle

ログイン画面を起点に、Artifacts（積み木）による実装と Playwright による比較検証を接続するための運用メモ。

## 目的

- AI 時代にふさわしい洗練感と視覚的訴求力を段階的に高める
- 実装の勢いを落とさず、毎回同じ手順で検証できるようにする
- 画面改善の判断を個人の感覚だけに依存させない

## 1サイクルの流れ

1. 対象画面を1つに絞る
2. その画面の改善テーマを1つ決める
3. 積み木へ渡す実装指示を `狙い / 変更要素 / 禁止事項 / 受け入れ条件` で書く
4. 実装後に `npm run test:visual` で desktop/mobile の比較を取得する
5. スクリーンショットと実画面を見て採否を決める
6. 採用した判断は screen spec に反映する

## レビュー観点

各サイクルで以下を 5 段階で評価する。

- 先進感: AI ネイティブな印象があるか
- 独自性: ありきたりな SaaS 画面に見えないか
- ブランド一貫性: タイポグラフィ、色、余白、モーションが揃っているか
- 視認性: 見出し、本文、CTA の優先順位が明確か
- 操作開始のしやすさ: 初見で何を押せばよいか迷わないか

## 実装メモ

- 初回対象は `/login`
- baseline 取得は `npm run test:visual:update`
- 通常確認は `npm run test:visual`
- 比較対象は full page と primary CTA の2種類
- 1回の改善でロジック変更とデザイン変更を混在させない

## 積み木向け入力テンプレート

```md
対象画面: /login
狙い: AI時代らしい先進感を強め、初見で押したくなる導線にする
変更要素: 背景、ヒーローコピー、CTA、補助カード、ローディング見た目
禁止事項: OAuth の処理フロー変更、ルーティング変更、過度な装飾で可読性を落とすこと
受け入れ条件:
- ファーストビューでCTAが見える
- mobileで横スクロールが出ない
- 背景演出が文字可読性を損なわない
- Playwright の baseline 比較を更新できる
```
