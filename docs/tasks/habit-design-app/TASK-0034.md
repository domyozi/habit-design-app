## A. クライアント側ストレージとトークン保護

### A-1. Notes / Coach context を `sessionStorage` に移行（既存 #1）
- **対象**: `BusyBoy2/src/lib/notes/useNotes.ts`、`src/lib/useCoachContext.ts`
- **方針**: localStorage → sessionStorage（タブ閉じで消える）。XSS 経由の長期窃取面を縮小
- **トレードオフ**: 体感は遅くなる（毎回 API 待ち）

### A-2. Notes を IndexedDB + Web Crypto で暗号化保管（既存 #5）
- **対象**: 同上
- **方針**: ノート件数が数百以上 / 機密度高いノートが想定されたら必須
- **方式**: Web Crypto AES-GCM、鍵を session 単位で derive

### A-3. localStorage の Supabase auth token を httpOnly cookie auth に置換
- **方針**: Supabase v2 の `flowType: 'pkce'` + cookie storage 設定
- **効果**: XSS による token 窃取を構造的に不可能にする
- **注意**: backend / FE 両方の auth handling 改修が必要

### A-4. `MarkdownText` に DOMPurify 多重防御（既存 #2）
- **対象**: `BusyBoy2/src/components/flow/MarkdownText.tsx`
- **方針**: `react-markdown` 既定 + 今回入れた urlTransform に加え、HTML 出力後に DOMPurify を通す保険

## B. 認証・OAuth 強化

### B-1. Google OAuth direct flow に PKCE 実装
- **対象**: `backend/app/services/google_token_service.py`、`backend/app/api/routes/google_oauth.py`
- **方針**: Authorization Code Flow に `code_challenge` / `code_verifier` を追加
- **発火条件**: ネイティブアプリ展開 / セキュリティ要件が上がった段階

### B-2. JWT 検証で `iss` を明示確認
- **対象**: `backend/app/core/security.py`
- **方針**: 現状 `audience="authenticated"` のみ強制。`iss` (Supabase URL) の明示検証で改ざん耐性を上げる

### B-3. critical 操作の再認証
- **対象**: アカウント削除、Google 連携解除、profile bulk update
- **方針**: 直近 N 分以内の認証が必要、または再認証チャレンジを挟む

## C. AI / プロンプト安全性

### C-1. Anthropic API のデータ取扱の明示確認 + 設定
- **方針**: Anthropic Trust Center の "data privacy" 規約を運営として確認 → 必要に応じて contractual opt-out 締結
- **デフォルト**: Anthropic API は学習に使われない（API 経由 = no training）

### C-2. Calendar 連携のプライバシーモード
- **対象**: `backend/app/services/coach_prompts.py` の `_calendar_section`
- **方針**: 予定タイトル全文を prompt に渡すのを「予定あり/なし」「カテゴリのみ」のような抽象化モードに切替可能にする
- **UX**: ユーザーが Settings で選択可能（プライバシー優先 vs パーソナライズ優先）

### C-3. AI 月次 cap（コスト × abuse 対策）
- **対象**: `backend/app/api/routes/coach.py` `/coach-stream`
- **方針**: per-user で月 N 回まで（無料枠）、超過時は 429 で拒否 or 課金枠誘導
- **実装**: `coach_pending_actions` count で集計、または別 usage table を設ける

### C-4. tool_use (`web_search`) 結果の sanitize
- **対象**: `backend/app/services/ai_service.py`
- **方針**: web_search 経由で外部サイト → AI へ injection 経路を狭める。tool result も sanitize / 検証

## D. インフラ・運用

### D-1. 監査ログ（既存 #4）
- **対象**: `backend/app/main.py` middleware or `audit_logs` テーブル新設
- **方針**: 全 mutation の (user_id, route, timestamp, request 概要) を別テーブルに記録
- **発火条件**: 有料ユーザー対応 / GDPR 規制 / インシデント調査

### D-2. Vercel preview と本番 Supabase の分離
- **方針**: Supabase の `branch` 機能で preview 環境を分離（Pro 以上）
- **効果**: PR preview から本番 DB を触れる事故を構造的に防止

### D-3. Backup / PITR（Point-In-Time Recovery）
- **方針**: Supabase Pro の PITR 設定を有効化、Free 期は日次 backup を確認
- **手順**: 復元手順を docs 化

### D-4. CSP の script-src を nonce / hash で厳格化（既存 #6）
- **方針**: 現状 `script-src 'self'` を `'self' 'nonce-XXX'` に。`'unsafe-inline'` 系の余地を全部潰す
- **コスト**: vite ビルドで nonce 注入の仕組みが必要

### D-5. Connect-src ワイルドカードの厳格化
- **対象**: `BusyBoy2/vercel.json`
- **方針**: `*.railway.app` / `*.supabase.co` を具体的なホスト名に絞る
- **コスト**: 5 分

### D-6. Frontend Supabase anon key の権限最小化（既存 #3）
- **方針**: Supabase コンソールで anon key の row/column 制限を厳格化

### D-7. OAuth トークンを Cloud KMS に逃がす（既存 #7）
- **方針**: 環境変数 Fernet 鍵 → GCP KMS / AWS KMS / HashiCorp Vault
- **発火条件**: 連携ユーザー数十人以上 or 社内ポリシー要求

### D-8. SSE stream integrity
- **対象**: `backend/app/api/routes/coach.py` `/coach-stream`
- **方針**: stream に message ID / sequence number を付与、FE で dedup
- **発火条件**: re-connect / replay 攻撃が疑われる規模

## E. データ主体の権利（GDPR / 個人情報保護法）

### E-1. アカウント削除エンドポイント + UI
- **対象**: backend `/api/users/me` DELETE、Settings に「アカウント削除」ボタン
- **動作**: `auth.users.id` 削除 → 各テーブル ON DELETE CASCADE で関連データ全削除
- **UX**: 復旧不可の警告 + 確認ダイアログ
- **発火条件**: **MVP 公開 = ほぼ必須**（特に EU/CCPA 圏向けに公開する場合）

### E-2. データエクスポート機能
- **対象**: backend `/api/users/me/export` で JSON dump
- **方針**: ユーザーが自分のデータを download できる "Right to Access"

### E-3. プライバシーポリシー / 利用規約の法務レビュー
- **対象**: `BusyBoy2/src/pages/PrivacyPage.tsx`、`TermsPage.tsx`
- **状態**: 暫定版を実装済。法務レビューで GDPR / 個人情報保護法 / Anthropic Terms との整合性を確認

### E-4. COPPA / 13歳未満対応

## F. ベータ運用 / ユーザー集客フロー（β 期）

### F-1. SNS 集客 → 事前登録 → Test users 追加 → 案内メール
**背景**: Google OAuth が「Closed Beta（Test users 制）」のため、誰でもサインアップできる構造ではない。
SNS（X/Twitter 等）から興味を持った人が「使ってみたい」と思った時の動線を整える。

**推奨フロー（〜100 人想定）**:

```
SNS 投稿
  ↓ ランディングページ（β登録フォーム）
  ↓ Email 入力（+ 任意で X handle / 用途）
  ↓ 登録 → 自動で「ウェイトリスト登録ありがとう」メール
  ↓ 運営者がフォームを確認
  ↓ Google Cloud Console「Test users」に手動追加
  ↓ 自動 / 手動で「招待が完了しました」メール送信
  ↓ ユーザーが busy-boy2.vercel.app → Google サインイン → ✓
```

**実装オプション** (低 → 高コスト順):

| 案 | 実装 | コスト | スケール |
|---|---|---|---|
| **A. Google Forms + 手動運用** | Google Forms で email 収集 → 通知メール → 運営者が Test users 追加 → Gmail で送信 | 0 円 / 30 分 | 〜30 人 |
| **B. Notion / Tally + Resend** | Tally form → webhook で Notion DB → 運営者がレビュー → Resend API で template メール | 0〜数千円 / 2 時間 | 〜100 人 |
| **C. Vercel ランディング + 自前 backend** | `/waitlist` エンドポイントを backend に追加 → Supabase に `beta_signups` テーブル → 運営ダッシュボード→ Resend 送信 | 半日〜1 日 | 〜数百人 |

**推奨**: 規模 ≦100 人なら **A or B**。

### F-3. β 期の ops チェックリスト

- [ ] SNS 投稿で Daily.OS の存在を周知（X/Twitter, Threads etc）
- [ ] ランディングページ or 登録フォームを設置（A〜C 案から選択）
- [ ] 登録 email を毎日 / 隔日でレビュー → Test users 追加
- [ ] 招待メールを送信（template F-2）
- [ ] 90〜100 人到達時に「申し訳ない、満員」へ form 切替（次のフェーズ準備）
- [ ] 90 人到達したら Google OAuth verification 申請の準備開始（独自ドメイン化、scope justification 文書、demo 動画）

### F-4. β を抜けて Verified Production へ移行（次フェーズ）
**前提条件**:
- 独自ドメイン（例: app.vekto.jp）で本番運用に切替（vercel.app は verification 通らない傾向）
- Privacy / Terms ページの法務レビュー完了
- App ロゴ完成

**手順**:
1. Google Search Console でドメイン所有確認
2. OAuth consent screen を全項目記入（domain / privacy URL / terms URL / logo / contact）
3. Sensitive scope（calendar.events / readonly）の justification 文書
4. デモ動画 (YouTube unlisted) 撮影 — Calendar 連携の動線を映す
5. 「Submit for verification」
6. 2〜6 週間の Google review
7. 通れば一般公開（警告なしでログイン可）

**ヒント**:
- verification 失敗のよくある原因: privacy policy の記載不足 / domain 所有不一致 / scope justification が曖昧
- 公開時には ToS / Privacy の **更新日** を最新化、新規ユーザー向けに同意取得 UI を追加
