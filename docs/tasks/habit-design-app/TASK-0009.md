## タスク概要

ユーザーの自然言語テキスト（音声入力を想定）をClaude APIで分類し、チェックリスト/ジャーナリング/日報/KPI更新/不明のいずれかに振り分けるサービスを実装する。チェックリストに分類された場合は習慣ログを自動更新する。Claude API障害時はトラッキング機能を継続する（EDGE-001）。

## 実装詳細

### 1. 音声分類サービス（voice_classifier.py） 🔵

**信頼性**: 🔵 *REQ-401/402/403・NFR-101 より*

Claude APIにテキストと習慣リストを送信して入力内容を分類する。

#### `classify_voice_input(text: str, user_habits: List[Habit], date: date) -> ClassificationResult`

送信データの制限（REQ-605）:
- 習慣タイトル（`habit.title`）のリストのみ送信
- ユーザーの個人情報（メール、ID等）は送信しない

プロンプト設計:
- システムプロンプト: 分類ルール（チェックリスト/ジャーナリング/日報/kpi_update/unknownの定義）
- ユーザーメッセージ: テキスト + 習慣タイトルリスト
- 出力形式: JSON（`type`, チェックリスト時は各習慣の達成/未達成フラグ）

分類結果の型（`ClassificationResult`）:
```python
class ClassificationResult:
    type: Literal["checklist", "journaling", "daily_report", "kpi_update", "unknown"]
    habit_results: Optional[List[HabitCheckResult]]  # checklist時のみ
    content: Optional[str]  # journaling/daily_report時のテキスト
```

`HabitCheckResult`:
```python
class HabitCheckResult:
    habit_id: str
    habit_title: str
    completed: bool
    confidence: float  # 0.0-1.0
```

Claude API障害時の処理:
- `anthropic.APIError` をキャッチ
- `AIUnavailableError` をraiseして呼び出し元でハンドリング

**実装ファイル**: `backend/app/services/voice_classifier.py`

---

### 2. 音声入力APIエンドポイント（voice_input.py） 🔵

**信頼性**: 🔵 *REQ-401/402・api-endpoints.md より*

#### `POST /voice-input`

リクエスト: `VoiceInputRequest`（text, date）
- `date`: 省略時はユーザーのタイムゾーンでの今日

処理フロー:
1. ユーザーの有効な習慣一覧を取得（`GET /habits` と同等のクエリ）
2. `voice_classifier.classify_voice_input()` で分類
3. 分類結果に応じた後処理:

   **checklist**:
   - `habit_results` の各習慣について `PATCH /habits/{id}/log` 相当の処理を実行
   - ストリーク・バッジ更新も含める
   - レスポンスに更新されたログ一覧を含める

   **journaling** / **daily_report**:
   - `journal_entries` テーブルに `input_type=分類結果` でINSERT
   - レスポンスに保存された `JournalEntry` を含める

   **kpi_update**:
   - 現時点では `journal_entries` に保存（将来的にKPIテーブルへの連携を想定）

   **unknown**（EDGE-003）:
   - ログの保存は行わない
   - レスポンス: `{"message": "どの操作ですか？入力内容を確認してください。"}`

4. Claude API障害時（`AIUnavailableError`）:
   - ステータス503
   - レスポンス: `{"error_code": "AI_UNAVAILABLE", "message": "AIサービスが一時的に利用できません。通常のトラッキング機能は継続して使用できます。"}`（EDGE-001）

**実装ファイル**: `backend/app/api/routes/voice_input.py`

## 単体テスト要件

### テストケース1: チェックリスト分類の正常系 🔵

**信頼性**: 🔵 *REQ-401 より*

**Given**: ユーザーが「早起き」「筋トレ」の習慣を持つ（モックClaude API）
**When**: `classify_voice_input("早起き達成、筋トレはできなかった", habits, today)` を実行する
**Then**: `type="checklist"`、早起き `completed=true`、筋トレ `completed=false` が返る

---

### テストケース2: ジャーナリング分類の正常系 🔵

**信頼性**: 🔵 *REQ-402 より*

**Given**: モックClaude APIがジャーナリングとして分類するように設定
**When**: `classify_voice_input("今日は気分が良かった。集中できた。", habits, today)` を実行する
**Then**: `type="journaling"`、`content` にテキストが設定される

---

### テストケース3: 判断不能（unknown）時のレスポンス（EDGE-003） 🔵

**信頼性**: 🔵 *EDGE-003 より*

**Given**: モックClaude APIが unknown として分類するように設定
**When**: `POST /voice-input` に分類不能なテキストを送信する
**Then**: ステータス200と `{"message": "どの操作ですか？..."}` が返り、DBへの保存は行われない

---

### テストケース4: チェックリスト分類後のログ更新統合テスト 🔵

**信頼性**: 🔵 *REQ-401/403 より*

**Given**: ユーザーが「早起き」「筋トレ」の習慣を持ち、モックClaude APIがチェックリストに分類
**When**: `POST /voice-input` に `{"text": "早起き達成、筋トレはできなかった"}` を送信する
**Then**: 早起きの `habit_log.completed=true`、筋トレの `habit_log.completed=false` でDBに保存される

---

### テストケース5: Claude API障害時のエラー（EDGE-001） 🔵

**信頼性**: 🔵 *EDGE-001 より*

**Given**: Claude APIが `anthropic.APIError` を返すようにモック設定
**When**: `POST /voice-input` にリクエストを送信する
**Then**: ステータス503と `{"error_code": "AI_UNAVAILABLE"}` が返り、通常の習慣トラッキング機能は影響を受けない

---

### テストケース6: 個人情報のClaude API送信除外確認 🔵

**信頼性**: 🔵 *REQ-605・NFR-101 より*

**Given**: ユーザー情報（メール、ID）と習慣データがある
**When**: `classify_voice_input()` が Claude APIにリクエストを送信する
**Then**: Claude APIへのリクエストペイロードにユーザーID・メールアドレスが含まれないこと

## 注意事項

- Claude APIへの送信データは習慣タイトルと入力テキストのみ（REQ-605）。ユーザーID・メール・Supabase IDは絶対に送信しない
- Claude APIの出力はJSON形式で返すようプロンプトで指定する。JSONパースエラー時は `unknown` に分類する
- チェックリスト分類時に習慣タイトルのマッチングが曖昧になる場合がある（「筋トレ」と「ジム筋トレ」など）。Claude APIに習慣IDも含めた構造化出力を返してもらう設計が望ましい
- EDGE-001（Claude API障害）時も `/habits`（通常のトラッキング）は使用可能であることをレスポンスメッセージで明示する
