## 修正対象

### 1. APIパスの仕様統一 🔵

**現状**:
- `backend/app/api/routes/__init__.py` が `/api/v1` プレフィックスを採用
- `backend/app/api/routes/me.py` が `/me` を公開しており、実際の経路は `/api/v1/me`

**仕様**:
- `TASK-0005.md` は `/api` プレフィックス配下への統合を要求
- `api-endpoints.md` は `GET /users/me` を定義
- `TASK-0006.md` も `GET /users/me` / `PATCH /users/me` 前提で記述されている

**対応方針**:
- `/api/users/me` を正とし、共通ルーター構成を修正
- 必要なら互換パスの扱いを明示する

### 2. Supabase参照方式の仕様逸脱修正 🔵

**現状**:
- `backend/app/main.py` の lifespan は `init_supabase()` を呼ぶだけで `app.state.supabase` を設定していない
- `backend/app/core/supabase.py` はグローバルシングルトン参照のみ

**仕様**:
- `TASK-0005.md` に「`app.state.supabase` に格納」と明記されている

**対応方針**:
- lifespan で返却されたクライアントを `app.state.supabase` に保持
- エンドポイント側から `Request.app.state.supabase` で参照できる構造に揃える

### 3. 422エラーメッセージ日本語化の不足修正 🔵

**現状**:
- `backend/app/core/exceptions.py` は `"入力値エラー: body → name: Field required"` のように英語メッセージをそのまま含む

**仕様**:
- TASK-0005 完了条件に「422バリデーションエラー時に日本語メッセージが返ること」とある

**対応方針**:
- Pydantic の代表的なエラー種別を日本語へ変換する辞書を用意する
- 最低限、必須項目欠落・型不一致・文字数制約など主要ケースを日本語化する
