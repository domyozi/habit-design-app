## タスク概要

習慣の作成・取得・更新・論理削除（CRUD）APIを実装する。AI提案による習慣更新は `change_time / add_habit / remove_habit` の3アクションのみを許可し（REQ-303）、それ以外のAIアクションは `FORBIDDEN_ACTION` エラーで拒否する。習慣一覧には今日のログとWanna Be接続文を含める。

## 実装詳細

### 1. 習慣一覧取得（GET /habits） 🔵

**信頼性**: 🔵 *REQ-301・api-endpoints.md より*

`is_active=true` の習慣一覧を返す。

クエリパラメータ:
- `include_today_log: bool = true`: 今日の日付（ユーザーのタイムゾーン）に対応するログを含める

処理フロー:
1. `habits` テーブルから `is_active=true` かつログインユーザーの習慣を取得
2. `goals` テーブルをJOINして `goal.title` を取得
3. `include_today_log=true` の場合: `habit_logs` テーブルから今日のログをLEFT JOIN
4. 各習慣の `wanna_be_connection_text` を生成: `f"→ {goal.title}に +1"`（goal未設定時は空文字）

レスポンス: `APIResponse[List[HabitWithLog]]`（`HabitWithLog` は `Habit` に `today_log: Optional[HabitLog]` を追加したスキーマ）

**実装ファイル**: `backend/app/api/routes/habits.py`

---

### 2. 習慣作成（POST /habits） 🔵

**信頼性**: 🔵 *REQ-302・api-endpoints.md より*

新しい習慣をINSERTする。

リクエスト: `CreateHabitRequest`（goal_id, title, scheduled_time, frequency）

バリデーション:
- `title`: 必須・255文字以内
- `scheduled_time`: HH:MM形式
- `frequency`: `daily` または `weekdays`（月〜金）

処理:
1. `habits` テーブルにINSERT
2. `wanna_be_connection_text` はgoal_idが指定されている場合にgoal.titleから自動生成

レスポンス: `APIResponse[Habit]`（ステータス201）

**実装ファイル**: `backend/app/api/routes/habits.py`

---

### 3. 習慣更新（PATCH /habits/{habit_id}） 🔵

**信頼性**: 🔵 *REQ-303/304/305・api-endpoints.md より*

習慣を更新する。`action` フィールドで操作種別を判定する。

#### アクション種別

| action | 説明 | 許可 |
|---|---|---|
| `change_time` | AIが時刻を変更する提案 | 許可（REQ-303） |
| `add_habit` | AIが新しい習慣を提案 | 許可（REQ-303） |
| `remove_habit` | AIが習慣削除を提案 | 許可（REQ-303） |
| `manual_edit` | ユーザー手動編集 | 許可 |
| その他 | 上記以外のaction | 拒否（400 FORBIDDEN_ACTION） |

処理フロー:
1. `action` フィールドを確認
2. AI提案アクション（change_time / add_habit / remove_habit）または `manual_edit` 以外の場合: `FORBIDDEN_ACTION`（400）を返す
3. 対象習慣がログインユーザーのものでない場合: 403を返す（RLSによる自動拒否 + 明示チェック）
4. 更新処理を実行

`FORBIDDEN_ACTION` エラーレスポンス:
```json
{
  "error_code": "FORBIDDEN_ACTION",
  "message": "このAIアクションは許可されていません。使用可能なアクション: change_time, add_habit, remove_habit"
}
```

レスポンス: `APIResponse[Habit]`

**実装ファイル**: `backend/app/api/routes/habits.py`

---

### 4. 習慣論理削除（DELETE /habits/{habit_id}） 🔵

**信頼性**: 🔵 *REQ-306・api-endpoints.md より*

習慣を物理削除せず、`is_active=false` に更新する（論理削除）。

処理フロー:
1. 対象習慣がログインユーザーのものでない場合: 403を返す
2. `is_active=false` に更新
3. 関連する未来の `habit_logs` は変更しない（履歴保持）

レスポンス: 204 No Content

**実装ファイル**: `backend/app/api/routes/habits.py`

## 注意事項

- `FORBIDDEN_ACTION` チェックはスキーマレベル（`AIActionType`）とAPIレイヤーの両方で実施する。TASK-0005のスキーマ定義と整合性を保つこと
- `action` フィールドが未指定の場合は `manual_edit` として扱う
- `wanna_be_connection_text` は習慣取得時に動的生成してもよいが、`habits` テーブルに保存して更新する方が効率的
- 論理削除後も `habit_logs` の履歴はそのまま保持する。週次レビューの集計対象となる
