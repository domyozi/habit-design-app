## 実装詳細

### 1. 習慣ログ記録エンドポイント（PATCH /habits/{habit_id}/log） 🔵

**信頼性**: 🔵 *REQ-404/406・api-endpoints.md より*

習慣の達成・未達成を記録する。同日に複数回リクエストが来た場合はUPSERTで対応する。

リクエスト: `UpdateHabitLogRequest`（completed, note, log_date）
- `log_date`: 省略時はユーザーのタイムゾーンでの今日の日付（EDGE-102）

処理フロー:
1. 対象習慣がログインユーザーのものか確認（違う場合は403）
2. `habit_logs` テーブルに `(habit_id, log_date)` でUPSERT
3. `completed=true` の場合:
   - `streak_service.calculate_streak()` でストリーク計算・更新
   - `badge_service.check_and_award_badges()` でバッジ判定・付与
4. `completed=false` の場合:
   - `habits.current_streak` を0にリセット（REQ-503）
5. レスポンスに `streak`（更新後の連続日数）と `badge_earned`（付与されたバッジ、なければnull）を含める

レスポンス:
```json
{
  "success": true,
  "data": {
    "log": { ...HabitLog },
    "streak": 7,
    "badge_earned": { ...UserBadge } or null
  }
}
```

**実装ファイル**: `backend/app/api/routes/habits.py`（追加エンドポイント）

---

### 2. 未達成理由記録（POST /habits/{habit_id}/failure-reason） 🔵

**信頼性**: 🔵 *REQ-406・api-endpoints.md より*

習慣が未達成だった理由を記録する。週次レビューのAI分析で活用される。

リクエスト: `{"reason": "string", "log_date": "YYYY-MM-DD"}`

処理:
1. 対象習慣の所有者確認（違う場合は403）
2. `failure_reasons` テーブルにINSERT（同日同習慣でのUPSERTでも可）

レスポンス: `APIResponse[FailureReason]`（ステータス201）

**実装ファイル**: `backend/app/api/routes/habits.py`（追加エンドポイント）

---

### 3. ストリーク計算サービス（streak_service.py） 🔵

**信頼性**: 🔵 *REQ-501/502/503・EDGE-102 より*

連続達成日数を計算・更新するサービス。タイムゾーン考慮が重要な実装ポイント。

#### `calculate_streak(habit_id: str, user_id: str, log_date: date, timezone: str) -> int`

処理フロー:
1. `habit_logs` テーブルから `completed=true` のログを `log_date` 降順で取得
2. `log_date`（チェックした日）から1日ずつ遡って連続達成日数を計算
3. 途切れた日を見つけた時点でカウントを停止
4. `habits` テーブルの `current_streak` を更新
5. `current_streak > longest_streak` の場合は `longest_streak` も更新（REQ-502）
6. 計算した `current_streak` を返す

タイムゾーン考慮（EDGE-102）:
- `log_date` はユーザーのタイムゾーンでの日付（UTC変換は呼び出し元で実施）
- `pytz` または `zoneinfo`（Python 3.9+標準）を使用

**実装ファイル**: `backend/app/services/streak_service.py`

---

### 4. バッジ付与サービス（badge_service.py） 🔵

**信頼性**: 🔵 *REQ-901・api-endpoints.md より*

ストリーク達成に応じてバッジを付与するサービス。

#### `check_and_award_badges(user_id: str, habit_id: str, streak: int) -> Optional[UserBadge]`

処理フロー:
1. `badge_definitions` テーブルから `condition_type="streak"` かつ `condition_value <= streak` のバッジを取得
2. 各バッジについて `user_badges` テーブルを確認し、未取得かつ条件達成のものを特定
3. 対象バッジが存在する場合: `user_badges` にINSERT（重複防止: `(user_id, badge_id, habit_id)` にユニーク制約）
4. 付与したバッジ情報を返す（複数条件を同時達成した場合は最も高い条件のバッジを1件返す）
5. 条件を満たすバッジがない場合は `None` を返す

**実装ファイル**: `backend/app/services/badge_service.py`

## 単体テスト要件

### テストケース1: ログ記録・ストリーク更新の正常系 🔵

**信頼性**: 🔵 *REQ-501 より*

**Given**: ユーザーが習慣を持ち、前日チェック済みである（streak=1）
**When**: `PATCH /habits/{id}/log` に `{"completed": true}` を送信する
**Then**: ステータス200、`streak=2`、`badge_earned=null` が返る

---

### テストケース2: ストリーク途切れでリセット 🔵

**信頼性**: 🔵 *REQ-503 より*

**Given**: ユーザーの習慣が `current_streak=5` である
**When**: `PATCH /habits/{id}/log` に `{"completed": false}` を送信する
**Then**: `current_streak=0` にリセットされ、`longest_streak=5` は維持される

---

### テストケース3: streak_7達成時のバッジ付与 🔵

**信頼性**: 🔵 *REQ-901 より*

**Given**: `streak_7` のバッジ定義が存在し、ユーザーの `current_streak=6` である
**When**: `PATCH /habits/{id}/log` に `{"completed": true}` を送信する
**Then**: `streak=7`、`badge_earned` に streak_7 バッジの情報が含まれる

---

### テストケース4: バッジの重複付与なし 🔵

**信頼性**: 🔵 *REQ-901 より*

**Given**: ユーザーが既に streak_7 バッジを取得済みで `current_streak=6` である（一度途切れた後）
**When**: 再び streak=7 を達成する
**Then**: `badge_earned=null`（同一バッジは再付与されない）

---

### テストケース5: タイムゾーン境界値テスト（EDGE-102） 🔵

**信頼性**: 🔵 *EDGE-102 より*

**Given**: ユーザーのタイムゾーンが `Asia/Tokyo`（UTC+9）で、UTC時刻で日付が変わる直前（23:55 UTC = 翌日08:55 JST）に前日ログが記録されている
**When**: JST日付でストリーク計算を実行する
**Then**: JSTの日付基準でストリークが正しく計算される（UTC基準のズレが発生しない）

---

### テストケース6: 連続達成ストリーク計算 🔵

**信頼性**: 🔵 *REQ-501/502 より*

**Given**: habit_logsに5日連続 `completed=true` のログがある
**When**: `calculate_streak()` を実行する
**Then**: `current_streak=5`、`longest_streak=5` が返る

---

### テストケース7: 途切れた後のストリーク計算 🔵

**信頼性**: 🔵 *REQ-503 より*

**Given**: 7日前〜5日前は達成、4日前は未達成、3日前〜今日は達成のログがある
**When**: `calculate_streak()` を実行する
**Then**: `current_streak=3`（4日前で途切れているため）、`longest_streak=3` が返る

## 注意事項

- タイムゾーン処理（EDGE-102）はバグの温床になりやすい。`log_date` は常にユーザーのタイムゾーンでの日付（`date` 型）として扱い、UTC変換は認証レイヤーかミドルウェアで行う
- `habit_logs` へのUPSERTは `(habit_id, user_id, log_date)` のユニーク制約を前提とする。DBスキーマに制約が定義されていることを確認すること
- `longest_streak` は一度記録されたら小さな値で上書きしない。`current_streak > longest_streak` の場合のみ更新する
- バッジの重複付与防止はDBの `UNIQUE` 制約（`user_badges` テーブル）に依存するが、アプリケーションレイヤーでも事前チェックを行い、エラーログを残さないようにする
