## 実装詳細

### 1. Supabase SQL Editor での実行 🔵

**信頼性**: 🔵 *database-schema.sql より*

1. Supabase ダッシュボード → SQL Editor を開く
2. `docs/design/goal-kpi-tracking/database-schema.sql` の内容を全選択してコピー
3. SQL Editor に貼り付けて「RUN」を実行
4. エラーがないことを確認

### 2. 後方互換性の確認 🔵

**信頼性**: 🔵 *REQ-KGI-001 の設計方針より*

追加カラムは全て NULL 許容のため、既存の Goal レコードは影響を受けない。
以下のクエリで既存データが壊れていないことを確認する：

```sql
-- 既存の goals が NULL 許容カラムを持っていることを確認
SELECT id, title, target_value, target_date, metric_type
FROM public.goals
LIMIT 5;
-- → target_value, target_date, metric_type が全て NULL であることを確認
```

### 3. 新規テーブルと RLS の確認 🔵

**信頼性**: 🔵 *NFR-KPI-101 より*

```sql
-- RLS が有効か確認
SELECT tablename, rowsecurity
FROM pg_tables
WHERE tablename IN ('goals', 'kpis', 'kpi_logs', 'kpi_habits');
-- → 全テーブルの rowsecurity = true

-- kpi_logs の UNIQUE 制約確認
SELECT conname, contype
FROM pg_constraint
WHERE conrelid = 'kpi_logs'::regclass;
-- → kpi_logs_unique_per_day が UNIQUE として存在
```

## 単体テスト要件（SQL レベル）

### テスト1: goals カラム追加確認 🔵

**信頼性**: 🔵 *REQ-KGI-001 より*

```sql
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'goals'
  AND column_name IN ('target_value', 'current_value', 'unit', 'target_date', 'metric_type');
-- → 5行が返ること
```

### テスト2: metric_type 制約確認 🔵

**信頼性**: 🔵 *REQ-KGI-003 より*

```sql
-- 不正な metric_type の挿入がエラーになることを確認（DIRECT実行は不要、設計確認のみ）
-- goals_metric_type_check 制約が存在すること
SELECT conname FROM pg_constraint
WHERE conrelid = 'goals'::regclass AND conname = 'goals_metric_type_check';
```

