# Sprint Spec: habit-design-app ミニマムローンチ（wanna_be × AIマンダラチャート DB移行）

## 1. プロダクト概要

**一言説明**: ユーザーが「理想の姿（wanna_be）」とAI生成マンダラチャートをクラウドに保存し、端末やブラウザをまたいで継続利用できる。

**対象ユーザー**: 自分の長期ビジョンを構造化したいが、localStorageの揮発性に困っているアーリーユーザー。

**コアバリュー**: 現状すべてのデータがlocalStorageにのみ存在し、ブラウザをまたぐとデータが消える。この問題を解消してはじめて「ツールとして信頼できる」状態になる。

**成功の定義**: ログイン済みユーザーがwanna_beを入力してAIマンダラチャートを生成し、そのデータがSupabase DBに永続化され、再ログイン・別端末でも同じデータが復元されること。

## 3. スプリント計画

### Sprint 1: バックエンド基盤（マンダラAPI追加）

**目標**: `GET /api/mandala` と `POST /api/mandala` が動作し、Supabase DBへのデータ永続化が確認できる状態にする。

#### タスク
- [ ] F-01: mandala_chartsテーブル追加 — Supabaseダッシュボードまたはマイグレーションファイルでテーブルを作成し、RLSポリシーを設定する
- [ ] F-02: マンダラ保存API — `POST /api/mandala` がcellsをJSONBで受け取りDBに保存する
- [ ] F-03: マンダラ取得API — `GET /api/mandala` が最新のマンダラを返す。未登録時は204を返す

#### 完了基準（Evaluator が使う）
- 認証済みJWTでPOSTしたマンダラデータがDBに保存される
- 同じユーザーでGETすると保存済みのマンダラJSONBが返る
- 未登録ユーザーがGETすると204が返る
- 別ユーザーのマンダラはGETで取得できない（RLS）

---

### Sprint 2: フロントエンド DB連携切り替え

**目標**: WannaBeTabの全データ操作がDB経由になり、ページリロード後もデータが復元される。

#### タスク
- [ ] F-04: wanna_be保存のDB連携切り替え — 入力テキストを `POST /api/wanna-be/analyze` 経由で送信する（既存のSSEストリームを維持したまま）
- [ ] F-05: マンダラ生成後の自動保存 — `streamMandalaChart` のonDoneコールバック内で `POST /api/mandala` を呼び出す
- [ ] F-06: 初回表示時のデータ復元 — コンポーネントマウント時に `GET /api/wanna-be` と `GET /api/mandala` を呼び出し、取得データを画面に反映する
- [ ] F-07: マンダラ手動編集後の自動保存 — MandalaGridのonUpdateが呼ばれてから500ms後に `POST /api/mandala` を呼び出す（デバウンス）
- [ ] F-08: 未認証時のガード表示 — `useAuth` の `session` が null の場合、ログイン誘導UIを表示してAPIコールを防ぐ

#### 完了基準（Evaluator が使う）
- ページリロード後、以前入力したwanna_beテキストが入力欄または画面に残っている
- ページリロード後、以前生成したマンダラチャートが表示される
- マンダラ生成後にDBを確認するとmandala_chartsにレコードが存在する
- セルを手動編集してしばらく待つと、DBの値が更新される
- ログアウト状態でタブを開くと「ログインしてください」等のメッセージが表示される

---

### Sprint 3: クリーンアップ

**目標**: localStorage依存を完全に除去し、保存状態がユーザーに伝わる状態にする。

#### タスク
- [ ] F-09: ローカルフォールバックの削除 — `useLocalStorage('wannabe:mandala')` と `useLocalStorage('wannabe:goals')` の参照をすべて削除する
- [ ] F-10: 保存中インジケーター — `POST /api/mandala` の実行中・成功・失敗を画面に表示する

#### 完了基準（Evaluator が使う）
- `wannabe:mandala` キーがlocalStorageに書き込まれない
- マンダラ保存中にローディング表示が出る
- 保存失敗時にエラーメッセージが出てデータが失われない

## 4. ユーザーストーリー

### F-01: mandala_chartsテーブル追加

**As** バックエンド開発者
**I want** Supabaseにmandala_chartsテーブルが存在し、user_id・wanna_be_id・cells（JSONB）・created_at・updated_atカラムを持つこと
**So that** マンダラデータをユーザーごとに永続化できる

**受け入れ条件**:
- [ ] mandala_chartsテーブルが存在する
- [ ] RLSが有効で、自分のuser_idのレコードだけ読み書きできる
- [ ] cellsカラムがJSONB型である

---

### F-02: マンダラ保存API

**As** フロントエンド
**I want** `POST /api/mandala` に認証トークンとcellsデータを送れること
**So that** 生成・編集したマンダラをDBに保存できる

**受け入れ条件**:
- [ ] リクエストボディに `cells`（9×9グリッドのJSONオブジェクト）を渡せる
- [ ] 既存レコードがある場合はupsertされ、重複レコードが作られない
- [ ] 認証なしのリクエストは401を返す

---

### F-03: マンダラ取得API

**As** フロントエンド
**I want** `GET /api/mandala` でDBに保存済みの最新マンダラを取得できること
**So that** ページを開いた直後に以前のマンダラを復元できる

**受け入れ条件**:
- [ ] 保存済みのcellsデータがレスポンスに含まれる
- [ ] 未登録の場合は204 No Contentが返る
- [ ] 認証なしのリクエストは401を返す

---

### F-04: wanna_be保存のDB連携切り替え

**As** ログイン済みユーザー
**I want** wanna_beを入力してGenerateボタンを押したとき、テキストがDBに保存されること
**So that** 次回ログイン時も自分のビジョンが残っている

**受け入れ条件**:
- [ ] Generateボタン押下時に `POST /api/wanna-be/analyze` が呼ばれる
- [ ] AIストリーミング表示は従来通り動作する（UX変化なし）
- [ ] localStorageへの書き込みは行われない

---

### F-05: マンダラ生成後の自動保存

**As** ログイン済みユーザー
**I want** AIがマンダラ生成を完了したとき、結果が自動でDBに保存されること
**So that** 手動で保存ボタンを押す必要がない

**受け入れ条件**:
- [ ] マンダラ生成完了後、`POST /api/mandala` が自動的に呼ばれる
- [ ] 保存失敗時もマンダラは画面に表示されたまま（データ消失なし）

---

### F-06: 初回表示時のデータ復元

**As** ログイン済みユーザー
**I want** WannaBeTabを開いたとき、以前保存したマンダラが自動で表示されること
**So that** 毎回生成し直す必要がない

**受け入れ条件**:
- [ ] ページ初期化時に `GET /api/wanna-be` と `GET /api/mandala` を呼び出す
- [ ] DBにデータがある場合はマンダラグリッドが表示される
- [ ] DBが空の場合は入力フォームが表示される

---

### F-07: マンダラ手動編集後の自動保存

**As** ログイン済みユーザー
**I want** グリッドのセルを編集したとき、変更がDBに保存されること
**So that** 編集内容がリロード後も保持される

**受け入れ条件**:
- [ ] セル編集から500ms以内に自動保存リクエストが発行される
- [ ] 連続編集時は最後の変更のみが保存される（デバウンス）

---

### F-08: 未認証時のガード表示

**As** 未ログインユーザー
**I want** WannaBeTabを開いたときにログインを促すメッセージが表示されること
**So that** APIエラーではなく分かりやすい案内を受け取れる

**受け入れ条件**:
- [ ] ログインしていない状態でタブを開くと、マンダラ入力フォームの代わりにログイン誘導UIが表示される
- [ ] APIコールは一切発生しない

---

### F-09: ローカルフォールバックの削除

**As** 開発チーム
**I want** `wannabe:mandala` と `wannabe:goals` のlocalStorageキーが使われなくなること
**So that** データの保存先がDBに一本化されシンプルになる

**受け入れ条件**:
- [ ] `useLocalStorage('wannabe:mandala', null)` の参照が削除されている
- [ ] ページ操作後にlocalStorageに `wannabe:mandala` キーが書き込まれない

---

### F-10: 保存中インジケーター

**As** ログイン済みユーザー
**I want** データが保存中・保存完了・保存失敗のいずれかを画面で確認できること
**So that** データが消えたか残っているか不安にならない

**受け入れ条件**:
- [ ] 保存中はインジケーターが表示される
- [ ] 保存完了後にインジケーターが消える
- [ ] 保存失敗時にエラーメッセージが表示される

## 7. 前提・依存関係と注意事項

### 既存APIの取り扱い

`POST /api/wanna-be/analyze` はSSEストリーミングを返す既存エンドポイント。F-04はこのエンドポイントを呼び出すだけで、バックエンド側の変更は不要。フロントエンドが `streamMandalaChart` 関数内で既に `streamClaude` を使っているため、ストリーミングUXは維持される。

wanna_beの保存はanalyzeエンドポイントが内部で行う設計（既存実装通り）。単純なテキスト保存だけを行う `POST /api/wanna-be` は現在のバックエンドには存在しないため、フロントエンドは `POST /api/wanna-be/analyze` を使う。

### mandala APIのupsert設計

`POST /api/mandala` はINSERTではなくupsert（1ユーザー1レコード）として設計する。wanna_be_idとuser_idの組み合わせをキーに、cellsだけを更新することで複数レコードの蓄積を防ぐ。

### JWTトークン付与

`frontend-v2/src/lib/api.ts` にはaxiosインターセプターが既に実装されており、全リクエストに `Authorization: Bearer` ヘッダーが自動付与される。フロントエンド側で追加の実装は不要。
